## CRITICAL

### C1. `reconcile()` 이중 체결 — 같은 종목·방향 주문이 둘 열리면 둘 다 '체결' 처리 (자금 장부 오염)
**파일:** `order-journal.js:176-177` (직접 확인)
```js
const filled = (e.side === 'buy'  && nowQty >= e.qtyBefore + e.qty)
            || (e.side === 'sell' && nowQty <= e.qtyBefore - e.qty);
```
각 `접수` 주문을 **현재 총보유량**에 대해 **독립적으로** 판정한다. `qtyBefore`는 주문 접수 시점의 보유 스냅샷.
- 보유 0 → 주문 A(buy 10, `qtyBefore=0`) → 잔고 갱신 전 주문 B(buy 10, `qtyBefore=0`).
- A만 실제 체결되어 보유 10 → reconcile: A는 `10>=0+10` 체결, **B도 `10>=0+10` 체결**. B는 실행된 적 없음.
- 매도는 더 위험: 두 매도 주문이 같은 `qtyBefore`를 잡으면 한쪽만 체결돼도 둘 다 체결 → 실현손익·잔여보유 모두 오기록.

**영향:** 체결 위조 → `pendingList` 왜곡(스테일 주문 자동취소 불가) → 엔진의 커밋자본·중복매도 가드 오판 → 실현손익 손상. VTS는 체결통보가 없어 reconcile가 유일 경로라 더 노출.
엔진의 종목당 1회 진입이 순수-봇 경로는 줄이지만, 멀티유저 수동매매·봇+수동 혼재에서 도달 가능.

**수정:** 주문별 절대판정 대신 **관측 델타를 소비**하도록. `(userId, code, side)`로 그룹화 → `min(qtyBefore)` 기준 총델타 계산 → **FIFO(id ASC)로 델타가 허용하는 만큼만** 채움. 최소판: 패스당 그룹별 **가장 오래된 1건만** 체결, 델타가 그 수량 이상일 때만.

### H2. 상태변경 엔드포인트 CSRF + localhost CORS 자격증명 반사
**파일:** `proxy-server.js:798-805`(CORS), `814-817`(쿠키), `/api/order`(2283)·`/api/cancel`(2394)·`/api/config`(1559)

**수정:** 상태변경 라우트에 커스텀 헤더 요구(`X-Requested-With`)+Origin 화이트리스트 또는 더블서밋 토큰. 운영에서 localhost CORS 반사는 비-production 플래그 뒤로.

### H4. KIS WebSocket 프레임 파서에 최대 크기 제한 없음 — 평문 ws:// 에서 OOM/행/크래시
**파일:** `kis-realtime.js:155-167`
```js
len = Number(this.buf.readBigUInt64BE(2));   // 상한 없음(최대 2^53)
...
if (this.buf.length < off + len) return;     // len 바이트 영원히 누적 대기 → 메모리 폭증
let payload = this.buf.slice(off, off + len);
```
KIS는 평문 `ws://`(TLS·핀닝 없음). 온패스/스푸핑된 프레임이 거대한 `len`을 선언하면 ① 프레임 영원히 미완 → `buf` 무한 누적 OOM, 또는 ② 거대 `.slice`/`.map` 언마스킹이 이벤트루프 정지·RangeError. `_drain`에 try/catch 없어 throw가 소켓 핸들러로 전파.

**수정:** `len` 계산 직후 상한(예 1 MiB)으로 `_fail()`; `_drain` 본문 try/catch → 재연결.

### H5. `.enckey` 읽기 시 권한 재설정 안 함 + 쓰기 실패 무음 → 마스터키 노출/무음 자격증명 전손
**파일:** `auth.js:53-61`
- `mode:0o600`은 **신규 생성 시에만**. 마이그레이션에서 `scp`/`cp`로 들어온 기존 `.enckey`가 0644여도 재-chmod 안 함 → 전 유저 KIS키를 푸는 마스터키가 world-readable 가능.
- 쓰기 실패를 `catch(e){}`로 삼킴 → 메모리 임의키 사용·미영속 → 재시작 시 **새 키 생성 → 기존 user-configs 전부 복호화 불가(무음 전손)**.

**수정:** 로드 시 `fs.chmodSync(KEY_FILE,0o600)` 방어; 쓰기 실패는 throw/경고; 읽은 키 길이 32바이트 검증.

### H6. `X-Forwarded-For` 최좌측 값 신뢰(tailscale 기본) → 인증 레이트리밋·잠금 우회
**파일:** `proxy-server.js:834-842`
`xff.split(',')[0]`(클라이언트 제공 최좌측)을 신뢰. 프록시가 append 방식이면 위조 `XFF`가 그대로 통과 — CF-Connecting-IP 때 겪은 동일 클래스 재발. 위조 IP 회전으로 로그인 5회 잠금(1523)·가입 10회/일(1496) 무력화, 또는 피해자 IP 위조로 표적 잠금(DoS).

**수정:** 신뢰 홉이 추가한 **최우측** XFF만, 또는 펀넬 전용 헤더만 신뢰. 보안 판단에 최좌측 XFF 금지. `_authGuard` 맵 상한도 함께.

### H8. 부분체결 미완결(VTS 영구 정체) + `markCancel`가 원주문 수량 파괴
**파일:** `order-journal.js:165-166`(reconcile는 `접수`만 스캔), `159-160`(직접 확인)
- `markFilled`가 `부분체결`로 바꾸면 reconcile(`접수`만 조회)이 영원히 완결 못 함 → VTS(체결통보 없음)에서 잔량 통보가 안 와 **영구 정체**, pendingList 부풀려 커밋자본·재매도 가드 오염.
- `markCancel`: `qty=fillQty`로 덮고 `canceledRemainder=1`(잔량 아닌 플래그) → **원주문 수량 소실**, "10주 주문/3체결/7취소" 감사 불가.

**수정:** reconcile가 `부분체결`도 완결 대상에 포함; `markCancel`은 원 `qty` 보존하고 `canceledRemainder = qty-fillQty`(스키마는 INTEGER인데 플래그로 낭비 중).

5. **재시작 후 중복 주문 억제** — 저널 pending 선적재 후 재기동 시 같은 종목 2차 매수 안 함.

## 확인된 정상(재보고 안 함)
- odno 충돌 userId-WHERE 수정(markFilled/markCancel) 정상.
- 합성 호가는 `_obCache`에만(synthetic:true), 엔진 가격경로(`_priceCache`/`_stockinfoCache`)와 분리 — 가짜가가 주문 유발 안 함.
- 정적 서빙 화이트리스트 — 레거시 HTML·`.enckey`·DB 도달 불가(경로순회 없음).
- SQL 전부 파라미터 바인딩(인젝션 없음). 가입 경쟁(BEGIN IMMEDIATE+admin 선출) 정상.
- scrypt 비동기+타이밍 동일화, AES-256-GCM(IV 랜덤·태그검증) 정상. 시크릿 git 미추적.
