# AutoTrade KR — 속도 재설계 제안서
### 모티브: 토스(Toss)급 즉각 반응성 / 우선 환경: 모의투자(VTS)

> 목적: "API 데이터가 사용자 입장에서 너무 느리다"는 문제를 구조적으로 해결한다.
> 범위: 지금 코드를 바로 고치기 전에, 무엇을 왜 어떤 순서로 바꿀지 합의하기 위한 설계 문서.

---

## 0. 한 장 요약 (TL;DR)

지금 느린 이유는 **데이터를 못 가져와서가 아니라, 모든 호출을 한 줄로 세워서**다.
서버에 전역 350ms 직렬 큐가 하나뿐이라 초당 약 3건이 한계인데, 이걸 **모든 사용자·모든 화면이 공유**한다. 관심종목 20개 시세를 한 번 채우는 데만 서버에서 `20 × 350ms ≈ 7초`가 걸리고, 화면 갱신 주기(4초)보다 느려서 큐가 계속 밀린다.

토스가 빠른 이유는 두 가지다. ① 화면을 **마지막 값으로 즉시** 그리고 뒤에서 갱신한다(기다리게 하지 않음). ② 시세를 REST 폴링이 아니라 **WebSocket으로 밀어준다**(요청-응답 왕복이 없음).

그래서 제안의 핵심도 세 줄이다.

1. **즉시 응답 계층** — 캐시된 마지막 값을 먼저 주고 뒤에서 새로고침 (stale-while-revalidate). 첫 화면 체감 0초.
2. **REST 큐를 사용자별로 분리하고 모드에 맞게 조인다** — 모의 5건/초, 실전 20건/초를 사용자마다 독립적으로 사용.
3. **KIS 실시간 WebSocket 도입** — 현재가/호가를 푸시로 받아 폴링 제거. 이게 토스급 즉각성의 본체.

모의투자 우선이므로, REST만으로는 초당 5건 한도 때문에 한계가 분명하다 → **WebSocket이 모의 환경에서 특히 효과가 크다.**

## 1. 현재 구조

```
브라우저(app.html)  ──REST 폴링(4초)──▶  proxy-server.js  ──HTTPS──▶  KIS REST API
   refreshLive()                          전역 350ms 큐              (모의/실전)
   refreshWatchPrices()                   pumpKis()
```

좋은 점은 이미 갖춰져 있다: 토큰 재사용, 5초 시세 캐시, 90초 거래량 캐시, 일봉 당일 캐시, EGW00201 백오프 재시도, 우선순위 큐(high/low). **틀은 잘 만들어져 있다.** 문제는 처리량(throughput)과 첫 응답(latency) 두 가지다.

## 2. 병목 진단 (수치 포함)

### 2-1. 전역 단일 직렬 큐 — 가장 큰 원인
`proxy-server.js`의 `KIS_MIN_GAP_MS = 350` + `pumpKis()`는 **전 사용자·전 엔드포인트**의 KIS 호출을 단일 큐에 350ms 간격으로 흘린다 (≈3건/초).

- `/api/prices`는 코드를 **서버에서 순차 루프**(line 732 `for (const code of codes)`)로 돈다 → 관심종목 20개면 `20 × 350ms ≈ 7초`.
- 화면은 4초마다 `refreshWatchPrices()`를 부르는데, 한 바퀴(7초)가 주기(4초)보다 길다 → **큐가 영구적으로 밀린다.**
- KIS rate limit은 **앱키(계좌) 단위**인데 큐는 전역이다 → 사용자가 2명만 돼도 각자 1.5건/초로 쪼그라든다. 한도를 낭비하는 구조.

### 2-2. 이중 스로틀
`/api/chart`, `/api/minchart`, `/api/market`, `fetchAccount` 등은 `rateLimitedCall`(또 다른 350ms 게이트, line 264)을 `kisProxy`(이미 큐 통과) **위에 한 번 더** 감싼다. 한 호출이 최대 700ms까지 늘어난다. `/api/prices`는 이미 제거했지만(line 736 주석) 나머지는 그대로다.

### 2-3. 분봉 14연속 호출
`/api/minchart`는 오늘 분봉을 위해 `times` 배열(line 963)로 14번 순차 호출 → `14 × 350ms ≈ 5초`. 게다가 과거 분봉은 일봉을 보간(`interpolateDayCandles`)한 **가짜 분봉**이라, 비용 대비 정확도가 낮다.

### 2-4. Cold start — 첫 화면에 보여줄 게 없음
첫 진입 시 토큰 발급 → 순차 시세 → 그 다음에야 숫자가 채워진다. 그 사이 화면은 비어 있다. 토스는 이 구간을 **마지막 값 + 스켈레톤**으로 메운다.

### 2-5. 폴링이라는 구조 자체
4초 폴링은 (a) 변화 없어도 매번 왕복하고 (b) 변화가 생겨도 최대 4초 늦게 보인다. 토스의 "값이 살아 움직이는" 느낌은 폴링으로는 안 나온다.

## 3. KIS 제약 (확인된 사실)

| 항목 | 모의투자(VTS) | 실전(LIVE) |
|---|---|---|
| REST 초당 호출 | **약 5건/초** | **약 20건/초** |
| 제한 단위 | 앱키(계좌)별 | 앱키(계좌)별 |
| 초과 시 | `EGW00201` 에러 → 재시도 필요 | 동일 |
| 실시간 WebSocket | 지원 | 지원 |
| WS 동시 구독 | 체결(H0STCNT0)+호가(H0STASP0) 합 **약 20종목**, 통보 포함 최대 ~41 | 동일 |
| WS 인증 | `approval_key`(`/oauth2/Approval`로 발급) | 동일 |

핵심 함의: **모의는 초당 5건이라 REST 폴링만으로는 관심종목 20개를 1초 안에 다 갱신할 수 없다.** 반면 WebSocket은 초당 한도와 무관하게 변동분만 푸시하므로 모의 환경에서 효과가 특히 크다. (모의용 WS 엔드포인트 동작 여부는 Phase 2 착수 시 실제 키로 1차 검증한다.)

## 4. 재설계 방향 — 3계층 데이터 파이프라인

```
┌─────────────────────────────────────────────────────────────┐
│ L1. 즉시 응답 (Instant)                                       │
│   - 마지막 값 메모리 캐시를 무조건 먼저 반환 (0ms 체감)        │
│   - 없으면 스켈레톤 → 채워지면 부드럽게 교체                   │
│   - stale-while-revalidate: 응답은 즉시, 갱신은 백그라운드     │
├─────────────────────────────────────────────────────────────┤
│ L2. 효율적 REST (Efficient pull)                              │
│   - 사용자(앱키)별 독립 큐 + 모드별 간격(모의 200ms/실전 60ms)│
│   - 서버 백그라운드 prefetch 루프가 관심종목을 미리 데워둠      │
│   - 차트/계좌 등 무거운 건 캐시 우선, 이중 스로틀 제거          │
├─────────────────────────────────────────────────────────────┤
│ L3. 실시간 WebSocket (Push) ← 토스급 즉각성의 본체            │
│   - 서버가 KIS WS 구독 → 브라우저로 SSE/WS 푸시                │
│   - 현재가·호가는 폴링 폐기, 변동분만 흘려보냄                  │
└─────────────────────────────────────────────────────────────┘
```

설계 원칙: **사용자를 절대 기다리게 하지 않는다.** 데이터가 없으면 마지막 값이나 스켈레톤을, 있으면 실시간 값을. 네트워크 왕복은 항상 화면 뒤에서 일어난다.

## 5. 단계별 실행 로드맵 (모의투자 우선)

각 Phase는 **독립적으로 배포 가능**하며, 앞 단계만으로도 체감이 좋아진다.

### Phase 0 — 즉효 패치 (반나절, 무위험)
지금 구조 유지, 큐만 손본다.
- 전역 큐 → **앱키별 큐**로 분리. (사용자 간 한도 간섭 제거)
- `KIS_MIN_GAP_MS`를 모드별로: 모의 `200ms`(≈5/초), 실전 `60ms`(≈16/초, 안전마진).
- `/api/chart`·`/api/minchart`·`/api/market`·`fetchAccount`의 **이중 스로틀 제거**.
- `/api/prices` 루프: 캐시 미스만 모아 큐에 한꺼번에 투입(현재도 직렬이지만 간격이 짧아져 20개 ≈ 4초 → 더 줄어듦).

→ **예상: 관심종목 첫 채움 7초 → 약 3~4초, 갱신 밀림 해소.**

### Phase 1 — 즉시 응답(L1) 도입 (1일)
- 모든 시세/계좌/지수 엔드포인트를 **stale-while-revalidate**로: 캐시값 즉시 반환 + 백그라운드 갱신.
- 서버 **prefetch 루프**: 로그인한 사용자의 관심종목 + 거래량 상위를 장중 주기적으로 미리 데움 → 사용자가 누르면 항상 캐시 히트.
- 프론트: 첫 진입 시 스켈레톤 → 캐시 → 라이브 순으로 부드럽게.

→ **예상: 첫 화면 체감 0~0.5초. "느리다"는 인상 대부분 해소.**

### Phase 2 — 실시간 WebSocket(L3) (2~3일) ★ 토스급 본체
- 서버: KIS `approval_key` 발급 → KIS WS 연결 → 사용자가 보고 있는 종목만 구독(H0STCNT0 체결, H0STASP0 호가).
- 서버 → 브라우저: **SSE(또는 WS)** 로 변동분 푸시. `refreshWatchPrices` 4초 폴링·`obTimer` 호가 폴링 폐기.
- 구독은 "현재 화면에 보이는 종목"으로 동적 관리 (WS 20종목 한도 대응 — LRU로 교체).
- 먼저 모의 키로 WS 엔드포인트 실제 동작을 검증한 뒤 확장.

→ **예상: 현재가·호가가 0.1~0.5초 단위로 살아 움직임. 폴링 트래픽·rate limit 압박 동시 해소.**

### Phase 3 — 마감 다듬기 (실전 키 발급 시점)
- 실전 키로 큐 간격 `60ms` 적용, WS 종목 한도·다계좌 분산 검토.
- 차트: 가짜 분봉 보간 정리, 필요한 봉만 지연 로드.
- 부하/재연결 안정화(WS 끊김 자동 복구, 백오프).

## 6. 프론트엔드 — 토스 UX 패턴

속도는 실제 속도 + 체감 속도의 합이다. 같은 데이터라도 다음이 "빠르게" 느껴진다.

- **스켈레톤 로딩**: 빈 화면 대신 회색 박스 → 값으로 교체.
- **낙관적/마지막값 우선 렌더**: 새 데이터 기다리는 동안 직전 값을 흐리게라도 보여줌(빈칸 0).
- **부드러운 숫자 전환**: 가격 변동 시 색 점멸·카운트업 애니메이션(이미 `up/dn` 클래스 보유 → 확장).
- **요청 합치기/취소**: 화면 전환 시 이전 화면의 진행 중 요청 취소(`AbortController`).
- **체감 0 우선순위**: 사용자가 방금 누른 종목은 `high` 우선순위로 새치기(이미 구조 있음 → 일관 적용).

## 7. 우선순위 / 예상 효과 요약

| 단계 | 작업 | 난이도 | 위험 | 관심종목 첫 채움 | 실시간성 |
|---|---|---|---|---|---|
| 현재 | — | — | — | ~7초 (밀림) | 4초 폴링 |
| Phase 0 | 큐 분리·간격·이중스로틀 | 낮음 | 무 | ~3~4초 | 4초 폴링 |
| Phase 1 | 즉시응답·prefetch | 중 | 낮음 | ~0초(캐시) | 4초 폴링 |
| Phase 2 | WebSocket 푸시 | 중상 | 중 | ~0초 | **0.1~0.5초** |
| Phase 3 | 실전 최적화 | 중 | 중 | ~0초 | 0.1~0.5초 |

권장: **Phase 0 → 1을 먼저 끝내 체감을 빠르게 끌어올리고**, 그 위에서 Phase 2(WebSocket)로 토스급 마감.

## 8. 리스크 및 검증

- **모의 WS 동작 불확실성**: 모의 환경 WS 지원 범위를 Phase 2 착수 시 실제 키로 먼저 검증. 안 되면 Phase 1(즉시응답+prefetch)만으로도 "느림"은 대부분 해결되므로 폴백 가능.
- **WS 20종목 한도**: "보이는 종목만 구독 + LRU 교체"로 대응. 관심종목이 많아도 화면에 보이는 건 제한적.
- **rate limit 재발**: 큐 간격을 모드별 안전마진으로. EGW00201 백오프는 이미 있음 — 유지.
- **회귀 검증**: 각 Phase 후 ① 첫 채움 시간 측정 로그 추가 ② 모의 키로 시세/호가/계좌/주문 정상 동작 확인 ③ 끊김·재연결 시나리오 테스트.
